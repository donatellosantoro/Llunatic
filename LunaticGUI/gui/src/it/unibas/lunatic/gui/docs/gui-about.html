1 Overview

This document describes Llunatic, a scalable, general purpose chase engine developed at the Università della Basilicata by the group lead by Prof. Giansalvatore Mecca. Llunatic supports data management problems where there is the need to exchange and clean data. In particular, given a declarative specification of the desired rules (possibly including a transformation between a source schema and a target one), Llunatic enforces them over the provided datasets with the semantics described in [2].

1.1 Description

The Llunatic system has been developed in Java using the NetBeans Platform as a basis for the graphical user interface. The system architecture is shown in Figure 1.


Figure 1: Architecture of Llunatic.

Llunatic supports three kinds of scenarios:

  • Type 1: schema-mapping scenarios in the spirit of data exchange literature, with one or more source databases, and a target database that is related to the sources by a set of schema mappings; integrity constraints can be imposed over the target; given a set of source instances, the goal is to generate a valid instance of the target according to the mappings.
  • Type 2: data repairing scenarios in the spirit of data quality literature, with one target database, possibly multiple authoritative source tables with highly curated data, used to model the so called master-data, and a set of data-quality rules over the target; these may include functional dependencies, conditional functional dependencies, editing rules, but also inclusion dependencies and conditional inclusion dependencies; here, we are given an instance of the target that is dirty with respect to the constraints, and want to generate a clean instance.
  • Type 3: mapping and cleaning scenarios, that combine and generalize the two above; here we have multiple sources that may include master-data, one target, a set of mappings that relate the target to the sources, and a rich set of data quality constraints over the target. Examples of such scenarios are in [2].

The above scenarios can be expressed by users in a scenario specification by using an XML file. The file describes several inputs, as described next. The file mentioned in the following are available in the examples folder. The complete list of options and parameters are reported in Section 5.

Data
Datasets can be defined in the xml file (e.g., treatments-poset-mcscenario) or can be loaded from an existing DBMS (e.g., treatments-poset-dbms-mcscenario). Master data are always defined as sources. Constants used in the dependencies are also represented in the sources as master data, as detailed in [1] (e.g., consttablecf1 in treatments-poset-mcscenario). Target data may contain data or be empty, but a target schema must always be provided as it contains the results of the process.
Dependencies
Target rules, such as data quality constraints, and source to target transformations are defined as logical expressions in the xml file. Supported formalisms are extended egds [1] and extended tgds [2].
Partial Order
In the xml file is possible to specify ordering attributes [1] (e.g., attribute customers.phone in the treatments scenario).

The scenario specification is then handled by the tool, which produces solutions for it. The number of alternative solutions and the interaction with the users are defined by using the cost and the user manager, respectively.

Cost Manager

It defines how to prune the chase tree and discard partial solutions along the process to limit the size of the output. The pruning is helpful both for scalability and for readability, as it reduces the solutions space and favor users consumption. Each cost manager is defined by a type and a set of configuration values.

Standard
It is the default cost manager. As it that keeps all solutions, it should be used for small scenarios only.
Forward-only (FO)
It accepts only forward repairs and discards nodes that contain backward ones. Use false for doBackward.
Frequency (FR)
It uses the frequency of values for an attribute to decide whether to forward or backward chase: values that are similar to the most frequent one are forward chased, the others are backward chased. In order to choose whenever two strings are similar, a strategy has to be defined, using two configuration attributes: similarityStrategy, one of the similarity metrics provided by SimMetrics (default is Levenshtein); and similarityThreshold, a value between 0.0 and 1.0 (default 0.8). A frequency cost manager is defined by a Frequency Partial Order and a Similarity Cost Manager. A complete example is in the scenario person-fr-sp-mcscenario.xml.

Standard, Forward-only and Frequency cost manager can be further customized by defining the maximum size of the chase tree: it accepts new branches in the chase tree as long as the number of leaves is less than the maximum size. There are two parameters to bound the chase tree size: chaseBranchingThreshold and potentialSolutionsThreshold. The difference between them is that the former considers all leaves, instead the latter discards from the counting duplicate and invalid leaves.

User Manager
It is used to decide when the chase should be stopped to request user input. Beside the standard one that never stops for user interaction, there are several strategies supported: (i) Interactive stops the chase and asks for inputs after each new node is added to the tree; (ii) AfterLLUN only stops for nodes that contain LLUNS; (iii) AfterFork stops the chase every time for than one child is created.

2 Specification Example

We now look in detail at the mapping and cleaning example in treatments-poset-mcscenario.

The source element specifies the schema to generate with a simple DTD-like syntax, followed by the instance. For an example with disk based execution (data loaded from a db) we refer the user to treatments-poset-dbms-mcscenario, where information for access-configuration are required. In these examples, there are two source tables for two target CFDs (e.g., consttablecf1 for cfd1) and only one tuple in the master data.

The target element specifies the target schema and its existing instance.

The dependencies element contains all the constraints of interest: the ones over the target schema (ExtTGDs and EGDs) to express the cleaning, and the source to target tgds to express the mapping (STTGDs). Notice how CFDs are defined by using the source tables storing their constants.

Finally, the orderingAttributes element contains two examples of attributes that are used in the partial order. When there is a problem for a phone, the correct preference is dictated by the values in cfphone. Similarly, for conflicts over salary, decisions can be taken by looking at the date (a recent salary is more likely to be correct).

For this example default options are used for several parameters. For an example of how to set the cost manager and the configuration, we refer the reader to treatments-usermanager-mcscenario. For an example of how to set the partial order and a different example of cost manager, we refer to person-fr-sp-mcscenario. For an example of user manager, we refer to doctors-usermanager-mcscenario.

3 Graphical User Interface and Main Functionalities


Figure 2: Screenshot of Llunatic GUI

Figure 2 depicts the main working screen for the system. The central panel opens a tab to show the content for an object (relation, cell group, or dependency) when it is double clicked from any of the other panels. The Property panel on the right shows information about the metadata of the objects when they are selected.

The button bar at the top contains four sections. The first section from the left has the basic operations to load and save a scenario. The second part contains undo and redo operations. The third part controls the chase with a button for complete execution and a button for step-by-step (Step by step is useful in two cases: (i) chase stopped because of the user manager, (ii) chase completed, but user changed a LLUN to a constant, thus triggering a new violation). The last button shows the results of the chase (cleaning result panel).

On the left side, the Cleaning result panel exposes all the solutions computed. Different icons are used for different nodes of the tree as follows.

There are two visualizations for the results, the chase tree or the list of solutions. Once a node of the tree is selected, the Step cell groups panels provides the information about the cells groups that have been modified in the step. At the bottom of the panel, nodes can be filtered by ground values/LLUNS (value filter, left) and by values that have changed or not (category filter, right). By double clicking a value in the cell group, its details are shown in the central panel.

At the bottom, the Dependencies panel shows/hides the rules for the current scenario. Double clicking a dependency opens a visual representation of the rule in the central panel.

On the right side, the first panel from the top contains the information about the Scenario, including the relations involved at the source and the target side. A single click on the scenario name shows its properties in the panel below: partial orders, cost managers, and other parameters. Double clicking a relation opens it in the central panel.

4 Advanced Features

4.1 User Interaction

The system implements user interaction to resolve LLUNS. To use this feature, users can right-click on a node in the tree, and then choose “Add User Node” so that they are able to edit cell groups with LLUN values. Once a LLUN is changed to a constant, all its occurrences are changed. By right clicking on a node, users can also invalidate chase trees: following executions will discard the node and its children. The level of interaction is defined via the User Manager.

One of the sample scenarios (doctors) uses a step-by-step user manager, which waits for user’s inputs after each LLUN is generated. Therefore users needs to progress the chase step-by-step.

4.2 Partial Order with JS

It is also possible to specify a custom JavaScript implementation of a partial order as part of the scenario, and implement it procedurally within the JavaScript code (e.g., treatmentsPartialOrder.js).
